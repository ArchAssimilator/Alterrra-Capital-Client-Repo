# Review of Sequential Prompt Chain, Version 1

## Overall Assessment

The structure is clear and the intent is strong, but Version 1 is not yet a reliable end-to-end process for recreating financial statements, rebuilding formulas, and turning them into a scenario-ready model. The biggest issue is that it conflates two different jobs:

1. Reconstructing historical statements from a source document
2. Building a forward-looking, assumption-driven forecast model

Those are related, but they are not the same workflow. Several prompts also assume spreadsheet capabilities that many GenAI environments do not actually have.

## 1. Errors and Mistakes in the Current Process

1. The overview claims the chain is "self-contained and executable in any GenAI environment", but later prompts require direct `.xlsx` creation, formula evaluation, workbook error scanning, dropdowns, cell locking, and conditional formatting. That is not true for many GenAI environments. References: `00 - Prompt Chain Overview.md:5`, `Prompt 3 - Reconstruct Formulas.md:43-50`, `Prompt 4 - Verify Formulas.md:21-47`, `Prompt 7 - Build Scenario Sheet.md:21-66`, `Prompt 8 - Verify Scenarios.md:48-52`.

2. The process jumps from historical table extraction straight into an "assumption-driven" and "scenario-enabled" model without an explicit forecast-build stage. Prompt 3 rebuilds formulas from reported historical tables, but Prompts 5 to 8 assume the existence of forward-looking driver logic. That model architecture is never actually created. References: `00 - Prompt Chain Overview.md:5`, `Prompt 3 - Reconstruct Formulas.md:18-41`, `Prompt 5 - Build the Assumptions Sheet.md:19-20`, `Prompt 5 - Build the Assumptions Sheet.md:70-77`, `Prompt 7 - Build Scenario Sheet.md:19-20`.

3. Prompt 5 asks the AI to derive base-case values for variables such as exit multiple, WACC, terminal growth, debt amortisation, and other deal-model drivers from the source data. Those values usually do not exist in the source financial statements and should not be inferred as if they were factual. References: `Prompt 5 - Build the Assumptions Sheet.md:40-56`, `Prompt 5 - Build the Assumptions Sheet.md:58-61`.

4. Prompt 5 instructs the AI to replace hardcoded inputs in the rebuilt model with assumptions-sheet references. That is unsafe if the model is still representing historical actuals. Historical actuals should remain immutable source-backed values; assumptions should drive forecast periods, not rewrite actual history. References: `Prompt 5 - Build the Assumptions Sheet.md:70-77`.

5. Prompt 7 contains a likely wiring flaw. It says the `Scenarios` sheet Base Case column should link to the `Assumptions` sheet, then says the `Assumptions` sheet should read from the `Scenarios` sheet Active Value column. That can create circular logic or make the Base Case column no longer represent a stable base case. References: `Prompt 7 - Build Scenario Sheet.md:36-42`.

6. The extraction format is too lossy for the stated standard of accuracy. Prompt 1 requires exact preservation of indentation hierarchy, merged-cell meaning, inline footnotes, blanks, and table structure, but stores everything in Markdown tables. Markdown is not robust enough for many complex financial statements, especially multi-level headers and merged structures. References: `Prompt 1 - Extract Financial Tables.md:17-35`, `Prompt 1 - Extract Financial Tables.md:43-47`, `Prompt 1 - Extract Financial Tables.md:55-66`.

7. Gate 1 is described as a "100% cell-by-cell match rate", but Prompt 2 only explicitly compares numerical cells. That means non-numeric structural issues can still exist even if the match rate is 100%. The gate definition is therefore stricter than the actual test being performed. References: `00 - Prompt Chain Overview.md:65`, `Prompt 2 - Verify Extraction.md:18-30`, `Prompt 2 - Verify Extraction.md:47`, `Prompt 2 - Verify Extraction.md:97-103`.

8. The tolerance rule "< 0.5 in the displayed unit" is not unit-aware. A tolerance that may be reasonable for currency amounts can be wildly inappropriate for percentages, ratios, EPS, per-share values, or leverage multiples. References: `Prompt 3 - Reconstruct Formulas.md:63-66`, `Prompt 4 - Verify Formulas.md:64`, `Prompt 8 - Verify Scenarios.md:25`.

9. Prompt 8 says to change each editable assumption by "+1 percentage point" during the wiring audit. That only makes sense for rate-based assumptions. It is incorrect for dollar values, unit counts, turns, dates, and other non-percentage variables. References: `Prompt 8 - Verify Scenarios.md:42-46`, `Prompt 8 - Verify Scenarios.md:124-125`.

10. Prompt 8 requires comparison of "every key output metric" against `01_Extracted_Tables.md`, but several listed metrics may not exist in the extracted source tables at all, such as Free Cash Flow, Debt / EBITDA, IRR, or MOIC. That makes the validation requirement partly impossible from the stated source of truth. References: `Prompt 8 - Verify Scenarios.md:21-33`.

11. Prompt 3 oversimplifies formula reconstruction by assuming reported subtotals, ratios, cross-sheet links, and items such as D&A can be safely derived from visible tables alone. In many real statements, these relationships depend on notes, hidden precision, presentation choices, eliminations, or supporting schedules that are not visible in the extracted tables. References: `Prompt 3 - Reconstruct Formulas.md:24-41`.

12. The stage-gate structure is slightly inconsistent. The overview says every stage has a gate, but Gate 3 is defined "after P5" even though Stage 3 includes both P5 and P6. P6 has no gate effect even though it sits inside the stage. References: `00 - Prompt Chain Overview.md:61-68`.

## 2. Areas for Improvement

1. Split the workflow into two explicit tracks: `historical reconstruction` and `forecast/scenario model build`. Historical reconstruction should end with a verified actuals workbook. Forecast modeling should begin only after drivers, forecast periods, and modeling architecture are defined.

2. Add a new stage between Prompt 4 and Prompt 5 for model architecture design. That stage should create forecast periods, operating schedules, working-capital logic, debt schedules, CapEx logic, balance-sheet balancing, and output summaries before any assumptions or scenarios are introduced.

3. Replace Markdown-only extraction with a more structured schema for tables. A better pattern would be Markdown for readability plus JSON/CSV for machine use, together with explicit metadata for merged cells, indentation level, units, footnotes, and source coordinates.

4. Add a strict sourcing policy for assumptions. Every assumption should be labelled as one of: `from source`, `derived from source`, `user provided`, `externally sourced`, or `analyst assumption`. Anything not present in the source should never be presented as if it were extracted fact.

5. Preserve historical actuals as locked source-backed values. Assumptions should feed forecast columns only. This will protect auditability and stop the workflow from rewriting actual reported numbers.

6. Redesign the scenario architecture so it cannot create circular references. The safest pattern is:
   - `Assumptions` stores static base inputs and metadata
   - `Scenarios` stores Base/Upside/Downside overrides
   - `Active Inputs` or a dedicated current-input range pulls from `Scenarios`
   - model formulas reference only the active-input range

7. Use type-specific validation tolerances. Example: exact match for labels and booleans, basis-point tolerance for percentages, small absolute tolerance for currency, and explicit rounding rules for per-share values and ratios.

8. Add an exception log for items the AI cannot reliably derive. This should include hidden precision, subtotal mismatch, note-dependent values, unsupported OCR sections, circular references requiring iteration, and assumptions that need human confirmation.

9. Reorder the human-input step. The questionnaire is useful, but it should feed the forecast/scenario build rather than appear as a side output after assumptions are already wired. In practice, user input should arrive before upside/downside scenarios are finalised.

10. Introduce environment-specific variants of the process. One version can be "AI text-only extraction and review". Another can be "spreadsheet-capable agent" that is allowed to create and test `.xlsx` files. Right now the prompts assume the second while claiming to support the first.

11. Add a clear stop condition for ambiguity. When the source does not support a formula, assumption, or relationship, the process should require the AI to mark it unresolved and escalate rather than infer a plausible answer.

12. Consider a standardized workbook architecture instead of "one sheet per financial table" only. A more durable structure would normally include `Source`, `Actuals`, `Assumptions`, `Schedules`, `Outputs`, `Scenarios`, and `Checks`.

## Recommended Next Revision Priority

1. Fix the Prompt 7 scenario-wiring logic.
2. Separate historical reconstruction from forecast modeling.
3. Add a dedicated forecast/schedule-building stage before assumptions and scenarios.
4. Replace Markdown-only extraction with a structured dual-format output.
5. Tighten sourcing rules so unsupported assumptions are never invented from historical statements.
